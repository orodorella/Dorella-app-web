## 1. Visión del Proyecto

### Qué es Dorela
Dorela es un ecosistema digital B2B/B2C para la venta mayorista y al detal de joyería en oro laminado 18k en Colombia. La plataforma sirve tres tipos de compradores — detal (público general), mayoristas y grandes mayoristas — cada uno con precios dinámicos calculados server-side según su tier. El sistema se integra con Alegra para facturación electrónica colombiana y gestión de inventario en tiempo real.

### Objetivos
- Automatizar el flujo de venta mayorista: pedido → factura → despacho
- Proteger precios mayoristas de exposición pública (nunca en HTML/API sin auth)
- Permitir pedidos en lote de 50-100 productos en menos de 2 minutos
- Gamificar la experiencia de compra con progreso hacia mejores descuentos

### Métricas de Éxito
- Tiempo para completar un pedido mayorista de 50+ items < 2 minutos
- 100% de órdenes facturadas automáticamente en Alegra
- Zero exposición de precios mayoristas a usuarios no autenticados

## 2. Tech Stack

| Capa | Tecnología | Por qué |
|------|-----------|---------|
| Frontend Framework | Next.js 15 (App Router) | SSR/SSG para SEO con 30-50k visitas diarias de pauta pagada. Server Components para rendimiento. Migración desde React+Vite SPA existente |
| Estilos | Tailwind CSS 4 | Ya implementado con tema custom — se migra tal cual |
| Animaciones | Framer Motion | Ya en uso — se mantiene en Client Components |

| Estado (servidor) | Server Components + fetch | Datos estáticos y catálogo se renderizan en servidor |
| Estado (cliente) | React Context + useReducer | Se mantiene para carrito, auth, toast — solo en Client Components |
| Backend API | Express.js + TypeScript | Se mantiene 100% intacto. Next.js lo consume como API externa |
| Base de datos | PostgreSQL (Supabase) | RLS para seguridad de precios, storage incluido, free tier |
| ORM | Prisma | Mejor DX, tipos auto-generados, migraciones confiables |
| Auth | JWT custom (bcrypt + jsonwebtoken) | Control total sobre lógica de tiers B2B |
| Validación | Zod | Validación tipada en API boundary |
| Jobs | BullMQ + Redis (Upstash) | Reintentos Alegra, PDFs, sync inventario |
| Storage | Supabase Storage | Imágenes productos, catálogos PDF |
| Email | Resend | API moderna, templates con React Email |
| Hosting API | Railway | Servidor persistente para BullMQ, deploy desde Git |
| Hosting Frontend | Vercel | Optimizado para Next.js: SSR, ISR, edge middleware, analytics |
| Package Manager | pnpm | Monorepo workspaces, más rápido que npm |

### Decisión Arquitectónica: Next.js como Frontend + Express como API Separada

```
┌─────────────────────────────────────────────────────────┐
│  VERCEL                                                  │
│  ┌─────────────────────────────────────────────────────┐ │
│  │  Next.js 15 (App Router)                            │ │
│  │                                                     │ │
│  │  Server Components ──fetch()──┐                     │ │
│  │  (SSR/SSG: Home, Catálogo,    │                     │ │
│  │   Producto, SEO pages)        │                     │ │
│  │                               ▼                     │ │
│  │  Client Components ──fetch()──┤                     │ │
│  │  (Carrito, Checkout, Admin,   │                     │ │
│  │   Auth flows)                 │                     │ │
│  │                               │                     │ │
│  │  middleware.ts ────────────────┤                     │ │
│  │  (Auth check, redirects)      │                     │ │
│  └───────────────────────────────┼─────────────────────┘ │
└──────────────────────────────────┼───────────────────────┘
                                   │ HTTPS
                                   ▼
┌─────────────────────────────────────────────────────────┐
│  RAILWAY                                                 │
│  ┌─────────────────────────────────────────────────────┐ │
│  │  Express.js API (sin cambios)                       │ │
│  │  /api/auth/*  /api/products/*  /api/orders/*        │ │
│  │  /api/admin/*  /api/webhooks/*                      │ │
│  │                                                     │ │
│  │  BullMQ Workers ──── Alegra, Email, Inventory       │ │
│  └─────────────────────────────────────────────────────┘ │
│  Redis (Upstash) ─── PostgreSQL (Supabase)              │
└─────────────────────────────────────────────────────────┘
```

**Por qué Next.js consume Express en vez de usar Route Handlers propios:**
- Express ya tiene toda la lógica de negocio, validación, pricing, Alegra — no duplicar
- Next.js se enfoca en rendering y SEO. Express se enfoca en lógica y datos
- Permite escalar API y frontend independientemente
- Los Server Components hacen `fetch()` al Express API server-side (sin CORS, baja latencia en misma red)
- Los Client Components hacen `fetch()` al Express API client-side (con CORS, JWT en header)

const API_INTERNAL = process.env.API_INTERNAL_URL;  // Railway internal URL (server-side)
const API_PUBLIC = process.env.NEXT_PUBLIC_API_URL;  // Railway public URL (client-side)

// Server-side fetch (en Server Components) — sin CORS, sin JWT para páginas públicas
export async function serverFetch(path: string, options?: RequestInit & { revalidate?: number }) {
  const { revalidate, ...fetchOptions } = options || {};
  const res = await fetch(`${API_INTERNAL}${path}`, {
    ...fetchOptions,
    next: revalidate !== undefined ? { revalidate } : undefined,
  });
  if (!res.ok) throw new Error(`API error: ${res.status}`);
  return res.json();
}

// Server-side fetch WITH auth (para páginas protegidas renderizadas en server)
export async function serverFetchAuth(path: string, token: string) {
  const res = await fetch(`${API_INTERNAL}${path}`, {
    headers: { Authorization: `Bearer ${token}` },
    cache: 'no-store',  // Datos personalizados, no cachear
  });
  if (!res.ok) throw new Error(`API error: ${res.status}`);
  return res.json();
}

// Client-side fetch (en "use client" components) — con JWT, CORS, refresh
export async function clientFetch(path: string, options?: RequestInit) {
  // Lógica de JWT + refresh se maneja en AuthProvider/useApi hook
  const res = await fetch(`${API_PUBLIC}${path}`, options);
  return res.json();

import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export function middleware(request: NextRequest) {
  const { pathname } = request.nextUrl;

  // Admin routes: verificar cookie de refresh token existe
  if (pathname.startsWith('/admin')) {
    const refreshToken = request.cookies.get('refresh_token');
    if (!refreshToken) {
      return NextResponse.redirect(new URL('/login?redirect=/admin', request.url));
    }
    // La verificación real del role admin ocurre en el layout de admin
    // (Server Component que hace fetch a /api/auth/me)
  }

  // Rutas protegidas de tienda
  const protectedPaths = ['/carrito', '/checkout', '/mis-pedidos', '/mi-perfil'];
  if (protectedPaths.some(p => pathname.startsWith(p))) {
    const refreshToken = request.cookies.get('refresh_token');
    if (!refreshToken) {
      return NextResponse.redirect(new URL(`/login?redirect=${pathname}`, request.url));
    }
  }

  return NextResponse.next();
}

## 8. Autenticación y Autorización

### Flujo de Auth

```
Registro → email/password → POST /api/auth/register → tier: detal → login automático
Login → POST /api/auth/login → { accessToken (15min), refreshToken (7 días) }
Refresh → POST /api/auth/refresh → nuevo accessToken
Logout → limpiar tokens del cliente
```

### Implementación JWT

```typescript
// Token payload
{
  sub: "user-uuid",
  email: "user@email.com",
  role: "cliente" | "admin",
  tier: "detal" | "por_mayor" | "gran_mayor",
  iat: 1719100000,
  exp: 1719100900  // 15 minutos
}
```

- **Access Token (15 min):** Se almacena en memoria del cliente (variable JS, NO localStorage). Se envía como `Authorization: Bearer {token}`.
- **Refresh Token (7 días):** Se almacena en httpOnly cookie con `Secure`, `SameSite=Strict`. Solo se envía a `/api/auth/refresh`.
- **Rotación de refresh token:** Cada refresh genera un nuevo refresh token e invalida el anterior.

### Middleware de Auth

```typescript
// middleware/auth.ts
// 1. Extrae JWT del header Authorization
// 2. Verifica firma y expiración
// 3. Inyecta req.user = { id, email, role, tier }
// 4. Si no hay token: req.user = null (acceso público con precios detal)

// middleware/requireRole.ts
// Wrapper: requireRole('admin') → 403 si req.user.role !== 'admin'
```

### Rutas Protegidas

| Patrón | Acceso |
|--------|--------|
| `GET /api/products` | Público (precios detal) / Auth (precios según tier) |
| `POST /api/orders` | Autenticado (cualquier tier) |
| `GET /api/orders` | Autenticado (solo sus órdenes) |
| `/api/admin/*` | Solo role: admin |
| `POST /api/auth/register` | Público |
| `POST /api/auth/login` | Público |

### Seguridad de Precios Mayoristas — Defensa en Profundidad

1. **Capa API:** `pricing.service.ts` calcula `precio_final` server-side. El campo `precio_base` NUNCA se incluye en respuestas a endpoints públicos ni de clientes. Solo `/api/admin/products` lo expone.
2. **Capa Base de Datos:** Row Level Security (RLS) en Supabase como segunda capa. Policy: `precio_base` solo legible para role `service_role`.
3. **Capa Frontend:** El frontend solo recibe y muestra `precio`. No hay lógica de descuento en el cliente.
4. **Rate Limiting:** 100 req/min para usuarios autenticados, 30 req/min para anónimos en `/api/products`. Previene scraping masivo.
5. **Logging:** Cada request a productos con tier > detal se registra para auditoría.

## 16. Reglas No Negociables

1. **`precio_base` NUNCA aparece en respuestas API a no-admin.** Test automatizado obligatorio que falle si se filtra. Esto es la regla más importante del sistema.
2. **Precios se calculan exclusivamente en Express `pricing.service.ts`.** Next.js recibe `precio` ya calculado. No hay lógica de descuento ni en Server Components ni en Client Components.
3. **Validación de mínimo de compra server-side en Express.** POST /orders rechaza con 400 si el total no alcanza el mínimo del tier. Next.js valida para UX, Express valida para seguridad.
4. **Stock con SELECT FOR UPDATE en transacción.** Evitar venta de productos sin stock por race conditions. Reservar stock al crear orden, liberar si se cancela.
5. **Alegra no bloquea el checkout.** La orden se confirma inmediatamente. La facturación en Alegra se procesa asincrónicamente via BullMQ. Si falla, retry automático con backoff exponencial (3 intentos). Failures se registran en `alegra_sync_log` para revisión manual.
6. **TypeScript strict en packages/api y packages/web.** `"strict": true` en ambos tsconfig. No usar `any` — usar tipos de packages/shared.
7. **Refresh token en httpOnly cookie SOLAMENTE.** Access token en memoria JS ref (NO localStorage). Refresh token con flags: `httpOnly`, `Secure`, `SameSite=Strict`, `Path=/api/auth/refresh`.
8. **CORS restringido.** Solo acepta requests de `CORS_ORIGIN` + preview deploys de Vercel. No usar `cors({ origin: '*' })` jamás.
9. **Rate limiting desde día 1.** `express-rate-limit` en todos los endpoints. Límites diferenciados: 30/min anónimo, 100/min autenticado, 10/min para login (anti brute-force).
10. **Server Components por defecto en Next.js.** Solo agregar `"use client"` cuando el componente usa hooks, event handlers, o browser APIs. Nunca poner `"use client"` en layouts o pages que no lo necesitan.
11. **ISR obligatorio en páginas de catálogo.** `/catalogo` y `/producto/[id]` deben usar `revalidate: 60`. Home usa `revalidate: 300`. Nunca `cache: 'no-store'` en páginas públicas indexables.
12. **Toda página pública exporta metadata SEO.** `generateMetadata()` o `metadata` estático con title, description, OG image. Sin excepciones para rutas bajo `(marketing)`.
13. **Express se mantiene intacto.** Next.js no duplica lógica de negocio. No usar Route Handlers de Next.js para lógica que ya existe en Express. Next.js es rendering + SEO, Express es datos + lógica.
14. **Las skills en `.claude/skills/` deben respetarse.** No sobrescribirlas ni ignorarlas durante la construcción.
15. **Cada request a la API Express devuelve el shape estándar:** `{ success, data, error, meta }`. Sin excepciones.
